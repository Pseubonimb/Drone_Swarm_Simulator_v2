# Серийные эксперименты и методика подбора коэффициентов PID

Документ суммирует **актуальное поведение** пайплайна «много прогонов подряд» в проекте и задаёт **план исследования** параметров регуляторов с **формальной скалярной метрикой** для сравнения прогонов.

---

## 1. Функционал выполнения множества экспериментов (актуальное состояние)

### 1.1. YAML-батч (`launch_simulation.py --batch-params`)

- За один запуск серии создаётся **папка сессии** с меткой времени старта:  
  `experiments/ГГГГ-ММ-ДД_ЧЧ-ММ-СС/` (четырёхзначный год).
- Каждый прогон пишется в подкаталог:  
  `experiments/<метка>/batch_<метка_из_YAML>_run_<n>/`  
  (`batch_run.json`, `metadata.json`, `drone_*.csv`).
- Повторный запуск той же серии **не перезаписывает** старые данные — появляется **новая** папка с новой меткой.
- Параметры SITL вида `sitl.ANGLE_MAX` при необходимости уходят в overlay-файл `.parm` в `experiments/_batch_parms/` с префиксом метки сессии, чтобы не конфликтовать именами между сессиями.

### 1.2. Скрипт `run_batch.py` с `--batch-id`

- Прогоны: `experiments/<ГГГГ-ММ-ДД_ЧЧ-ММ-СС>/batch_<id>_run_<n>/`.
- Индекс `batch_<id>_runs_index.json` лежит **в той же** папке сессии.

### 1.3. Связка SITL, MAVProxy и батча (рабочий вариант для корректных `drone_*.csv`)

- **Прямое TCP** к порту SITL (`--no-mavproxy`, `DRONE_SWARM_SITL_DIRECT_TCP`) в проекте предусмотрено как вариант для стабильных серий без «залипания» MAVProxy; при вашей конфигурации с ним **не гарантируется** приемлемая запись координат в `drone_*.csv`, поэтому для батча вы **его не используете**.
- Вместо этого серия запускается с **MAVProxy и консолями**, например:

  ```bash
  python launch_simulation.py -s --batch-params scenarios/batch_parameters/snake_pursuit_p_gain_batch.yaml \
    --with-mavproxy-consoles
  ```

  Лаунчер поднимает цепочку `sim_vehicle` с MAVProxy (UDP к сценарию), после **каждого** завершения прогона батча (в т.ч. при Ctrl+C в конце серии) вызывается **`pkill -f mavproxy.py`**, чтобы освободить порты и не мешать следующей итерации. Подробности и ограничения см. в `launch_simulation.py` (справка `--with-mavproxy-consoles`) и в `docs/found errors/06_04_26_errors_in_project.md` (п. 1 — почему без очистки MAVProxy батч ломается).

### 1.4. OFAT из командной строки при батче

- Если в YAML перебирается **не все** PID-оси, остальные коэффициенты можно зафиксировать **флагами родительского** `launch_simulation.py`:  
  `--kp`, `--ki`, `--kd`, `--derivative-alpha` — они **пробрасываются** в каждый дочерний прогон, **если** соответствующий логический ключ (`pid.p_gain`, …) **не** входит в текущую комбинацию sweep.
- Практические этапы и примеры YAML: каталог `ofat_snake_pursuit/` и `ofat_snake_pursuit/README.md`.

### 1.5. Плоттер по результатам батча (`plotter/plotter.py`)

- **По умолчанию** (без `--runs-glob`): берётся **последняя** по имени папка сессии под `experiments/`, имя которой совпадает с шаблоном `ГГГГ-ММ-ДД_ЧЧ-ММ-СС`, и строятся только прогоны  
  `experiments/<эта_метка>/batch_*_run_*`.
- Каталоги с двузначным годом в имени (`26-…`) в автовыборе **не участвуют**; при необходимости путь задаётся явным `--runs-glob`.
- Выходные PNG при однозначной сессии данных пишутся в  
  `<--out-dir>/<та_же_метка>/`  
  (имя подпапки совпадает с папкой сессии эксперимента), чтобы не смешивать графики разных серий.
- Режимы: `time_overlay` (динамика), `bar_metric`, `heatmap` (две оси параметров) — см. `plotter/README.md`.

### 1.6. Запись CSV и ожидание готовности SITL

- При `log_hz == 0` строки в лог координат пишутся **на каждом шаге** обмена (а не только при смене позы), иначе в POS_HOLD возможна **одна строка** в CSV. Дополнительно в коде увеличены таймауты старта и ожидания heartbeat, чтобы цикл обмена успевал заполнить лог — см. `docs/found errors/06_04_26_errors_in_project.md` (п. 4).

### 1.7. Конфигурация YAML: `batch:` и `launch:`

- Блоки **`batch:`** и **`launch:`** задают одни и те же поля сценария/длительности/дронов; **`launch:` перекрывает `batch:`** при совпадении ключей — удобно держать базу в `batch:` и точечные правки в `launch:`.

## 2. План действий по изучению коэффициентов (рекомендуемая последовательность)

Цель — **не** запускать полный факторный план по всем осям сразу (комбинаций экспоненциально много), а накапливать знания **пошагово**.

1. **Зафиксировать постановку**  
   Сценарий (`snake_pursuit` / `leader_forward_back`), число дронов, `duration_sec`, целевой зазор **d\*** (см. §3), допуск **ε**, файл параметров ArduPilot (`--param-file`). При необходимости **не менять** `ANGLE_MAX` на этапе настройки только PID.

2. **Воспроизводимость**  
   Один и тот же набор фиксированных настроек прогнать 2–3 раза (поле `iterations` в `experiment_settings`, если используется режим без конфликтующего `parameter_sweeps`), оценить разброс метрик.

3. **OFAT (one factor at a time)**  
   - Этап A: перебор только **P** (`pid.p_gain` → `--kp`), остальное — дефолты или флаги лаунчера.  
   - Этап B: зафиксировать лучший **kp\***, перебор **D** (`pid.d_gain`).  
   - Этап C: зафиксировать **kp\***, **kd\***, перебор **I** (`pid.i_gain` → `--ki`) на roll и pitch; при запуске батча **не** передавать родительский `--ki`, если он в sweep.  
   - Этап D: при необходимости перебор **`derivative_alpha`** (сглаживание D).  

   Готовые YAML: `ofat_snake_pursuit/stage*.yaml`, журнал: `tuning_results_template.csv`.

4. **Сужение интервалов**  
   После каждого этапа сузить диапазон вокруг лучшей точки (меньший шаг, меньше крайних значений).

5. **Двумерные сетки (опционально)**  
   Только **две** оси одновременно (например **kp × kd**), затем визуализация `heatmap` в плоттере или своя обработка `batch_run.json` + CSV.

6. **Цепочка из N дронов**  
   Сначала стабилизировать пару «лидер — первый фоловер», затем тот же закон на цепочку и смотреть накопление ошибки по звеньям; при необходимости — отдельная метрика «худшее звено» или сумма по всем парам.

## 3. Итоговая скалярная метрика **J** (нормальная запись и переменные)

Нужно сравнивать прогоны **одним числом** при фиксированных сценарии, горизонте моделирования **T** и целевом зазоре **d\***.

### 3.1. Вспомогательные величины по одному прогону

Пусть **d(t)** — выбранная вами дистанция между интересующими дронами (или отклонение от **d\***), восстановленная по времени **t** из `drone_*.csv` на отрезке **[0, T]**.

- **Целевой зазор d\***  
  Заданное вами эталонное расстояние (или эталонное отклонение), к которому должен прийти рой/пара. В `snake_pursuit` в коде заложен ориентир по поперечному смещению **≈ 2 m** между соседями в общей NED-сетке; в отчёте **d\*** формулируйте явно под вашу метрику (например норма в плоскости или только по оси Y).

- **Допуск ε (эпсилон)**  
  Положительное малое число: считаем, что «в полосе удержания», если  
  |d(t) − d\*| ≤ ε.

- **Время установления T\_ε**  
  Момент времени (первый подходящий **t**), после которого ошибка остаётся в полосе:  
  |d(t) − d\*| ≤ ε  
  (в отчёте уточните: первое вхождение, устойчивость не менее **Δt** секунд и т.д.).

- **IAE** (integral absolute error)  
  Интеграл модуля ошибки на горизонте **T**:  

  **IAE = ∫₀^T |d(t) − d\*| dt**  

  Штрафует долгое «ползание» к режиму даже при малом **T\_ε**.

- **Overshoot (перерегулирование)**  
  Скалярный штраф за выход за цель; примеры:  
  **max\_t |d(t) − d\*|**  
  или **max\_t d(t) − d\***, если важен только «пролёт» в сторону увеличения расстояния. В отчёте зафиксируйте выбранную формулу.

### 3.2. Сводная метрика **J**

Пример линейной свёртки (чем **меньше J**, тем **лучше** прогон при ваших весах):

**J = T\_ε + w₁ · IAE + w₂ · Overshoot**

где:

| Символ | Смысл |
|--------|--------|
| **J** | итоговый скалярный критерий качества одного прогона |
| **T\_ε** | время установления в полосу ±ε относительно **d\*** (секунды) |
| **IAE** | интеграл |d(t) − d\*| от 0 до **T** (м·с или согласованные единицы) |
| **Overshoot** | выбранная мера перерегулирования (те же единицы, что и d, или безразмерная нормировка) |
| **w₁** | неотрицательный вес штрафа за «долгую тягу» (IAE); чем больше **w₁**, тем сильнее вы требуете быстрого и плотного подхода к **d\*** по всему интервалу |
| **w₂** | неотрицательный вес штрафа за перерегулирование; чем больше **w₂**, тем сильнее вы штрафуете пики и «рывки» |

**w₁** и **w₂** подбираются под приоритет: «быстрее и точнее по траектории» (типично больше **w₁**) против «мягче, без больших выбросов **d**» (больше **w₂**). Можно один из весов обнулить, если критерий не используется.

Так **оптимальные настройки** регуляторов в рамках эксперимента формулируются как **минимум J** при фиксированных сценарии, **T**, **d\***, **ε** и правилах вычисления **Overshoot**, плюс жёсткие ограничения (столкновения, расходимость, пределы PWM и т.д.) — нарушение ограничения = прогон отбраковывается или получает **J = +∞** в таблице.

### 3.3. Связь с артефактами проекта

- В каждом прогоне: `batch_run.json` → словарь **params** (фактические **kp**, **ki**, **kd**, **derivative\_alpha**, … для этого run).  
- Временные ряды: `drone_1.csv`, `drone_2.csv`, … — для расчёта **d(t)** и интегралов.  
- Плоттер даёт обзорные графики; **T\_ε**, **IAE**, **Overshoot** для диплома удобно считать отдельным скриптом по CSV и заносить в `tuning_results_template.csv` (или расширенную таблицу).
